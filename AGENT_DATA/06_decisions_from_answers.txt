--------------------------------------------------------------------------------
1) DOWNLOAD LIMIT PER QR/LINK
--------------------------------------------------------------------------------
Answer: Configurable; display remaining count.

Implications:
  - Store per token: max_downloads (configurable), remaining_downloads.
  - Download page shows e.g. "2 downloads left" or "Already used (0 left)".
  - Backend must support configurable max when creating tokens (e.g. default 1 or 3).

--------------------------------------------------------------------------------
2) WHAT YOU ARE SERVING AS "THE ALBUM"
--------------------------------------------------------------------------------
Answer: ZIP is fine; it can contain all sorts of data.

Implications:
  - One ZIP file (MP3s, artwork, etc.) per album. Backend serves this file (or a signed URL to it) after token check.
  - Bandwidth: consider ZIP size and number of downloads for hosting limits (e.g. 100 GB/month on free tiers).

--------------------------------------------------------------------------------
3) HOW YOU WILL GENERATE QR CODES
--------------------------------------------------------------------------------
Answer: Initial bulk (e.g. 200 tokens), then option to generate more as needed (1 or 1000).

Implications:
  - Generator tool/script must support: (1) bulk generation (e.g. 200), (2) add N more tokens later (N = 1 to 1000).
  - Same script can output list of URLs for QR generation (node-qrcode). No backoffice; run locally when needed.

--------------------------------------------------------------------------------
4) USAGE TRACKING
--------------------------------------------------------------------------------
Answer: Prefer to know date and time when the link is exhausted (reached the limit).

Implications:
  - Store per token: exhausted_at (timestamp when remaining hit 0). Optional: first_used_at, last_used_at if you want more detail.
  - Optional: export (CSV/JSON) or read-only report listing exhausted tokens + exhausted_at. Can be a simple script that reads from your store (KV or file).

--------------------------------------------------------------------------------
5) HOSTING PREFERENCE
--------------------------------------------------------------------------------
Answer: Prefer free/cheap option.

Implications:
  - Use serverless (Netlify / Vercel / Cloudflare Pages) + KV or Blob store, not a paid VPS. See 07_hosting_suggestions.txt for concrete services.

--------------------------------------------------------------------------------
6) DOMAIN AND SSL
--------------------------------------------------------------------------------
Answer: SSL is a must; album.konaboj.cz was just an example (could be any address).

Implications:
  - Choose a host that provides free SSL (custom domain + HTTPS). All suggested hosts do this.

--------------------------------------------------------------------------------
7) EDITABLE CONTENT
--------------------------------------------------------------------------------
Answer: OK with editing HTML/React and redeploying.

Implications:
  - No CMS required. Static HTML/React; change copy, colours, images in code and redeploy. Keeps stack simple.
